## 一.人为什么会觉得某些东西美？

古语云：世人皆有爱美之心。但有多少人有思考过为什么人类会有美与丑的观念呢，为什么人会觉得某些东西美，某些东西丑？

### 1.1 美不是单一来源

美它并不是一个来源单一的判断，它可能同时来自人类祖先的进化积累产生的集体记忆，也就是荣格心理学中所说的集体潜意识，还有视觉加工、文化经验和个人记忆。

演化层面：有些美感可能与生存、繁衍、安全和资源判断有关，比如健康的面孔、清澈的水源(清澈的水源其实不一定安全，记得前几年好像就有游客喝湖水住院了)、开阔的视野，都可能在无意识层面让人联想到生命力、安全性和可居住性。

视觉加工层面：大脑更喜欢容易处理的信息。对称，清晰，稳定有规律的东西，往往更容易被大脑理解，因此也更容易被判断为舒服，自然，好看。

文化经验层面：不同文化会塑造不同的审美标准，比如西方教堂和中国寺院。某些图案，建筑风格，服饰，会在不同社会中赋予不同意义。

个人记忆：一个人童年喜欢的环境，熟悉的物品，重要的人和精力，会影响他的审美判断。

### 1.2 面孔吸引力

面孔的吸引力可能来自人类祖先对健康、发育稳定性和生命力的敏感。

从演化心理学的角度看，人类对面孔美感的判断，可能与平均性，对称性，健康感等线索有关。但这并不意味着”美貌=优秀基因“，只是大脑可能会把某些外部特征当做快速判断的线索。并且这个判断过程往往是难以察觉的，因为我们往往很难察觉自己的潜意识，人的表意识处理信息大约只有10比特每秒，而潜意识(更准确的来说是感官系统的处理速度)差不多能达到10^9bit/s，是表意识的差不多一亿倍。

在意识形成判断之前，感官系统和无意识加工已经完成了大量筛选、压缩和预判。很多时候，我们进入某个地方后突然觉得“不安全”“不舒服”“哪里不对劲”，可能不是玄学，而是大脑在无意识层面捕捉到了光线、空间、声音、人群状态等信息，只是这些信息还没有被清晰地翻译成语言。

### 1.3 自然风景：安全栖息地偏好

自然界美丽的风景之所以让我们感到美丽，可能与人类祖先对安全栖息地的偏好有关。

人类并不是随机地喜欢自然景观。开阔视野意味着更容易发现机会与危险，水源意味着资源，树木和遮蔽物意味着安全。在UIrich 1984一个关于医院窗景的经典研究中发现，能看到自然景观的术后患者，相比于面对砖墙景观的患者，祝愿时间更短，强效止痛药使用更少。

这不是说美好的风景可以治疗疾病，只是说明自然环境会影响人的激素，进而影响到情绪(其实一开始我打算把情绪放到激素前的，不过这样有些颠倒因果了)。这说明人会对环境中的安全感，开放感和生命感产生反应。

所以，当我们觉得一片自然风景很美丽时，可能不只是因为它好看，而是它在潜意识或者说无意识层面暗示了资源，安全和生命力。

### 1.4 界面之美：加工流畅性

对面孔和自然风景的美感一般来自于演化和文化这些，那么界面中的美感很大一部分来自一个重要概念：”加工流畅性“

先说一下加工流畅性的定义：一个界面越容易被大脑处理，人们就越容易对它产生积极的感受。

如果一个界面元素很多，颜色混乱，按钮样式不统一，文案含糊不清，用户就要不断消耗注意力去理解这个界面。即使功能设计这些本身没有什么问题，用户也有可能产生排斥，觉得这个东西复杂难用的心理。格式塔心理学也涉及这些，关于用户如何自动组织视觉信息。

## 二.格式塔心理学

格式塔心理学中有一个很重要的观点：整体不等于部分的简单相加。

这个听起来是不是有点像涌现的概念，整体>部分之和，而这个多出来的部分就是涌现。

其实我们看界面也是这样，并不是一个个读取每一个按钮，图片，文字和图标。人的大脑会自动寻找结构，把相近的元素看成一组，把相似的元素看成一类，把被边框包裹的内容看成同一个区域，把没有完全闭合的图形补全成完整形状。

或许人类很难看出在绿色植物中有保护色的动物也是因为这个，人脑会按区域来看，进行一个视觉的组织。(原本想在绿色植物这里举两个例子，但想想觉得这两个例子或许有些吓人了)

这对设计非常重要，用户理解界面的过程是从第一步的视觉组织开始的。

格式塔心理学中有几个原则

- 接近性
- 相似性
- 共同区域
- 闭合
- 简洁律：复杂不是问题，无序才是问题

### 2.1 接近性

接近性原则指的是视觉上离得更近的元素，会被人认为关系更密切；距离更远的元素，会更容易认为属于不同分组。

比如在一个表单中，如果用户名标签离离输入框太远，用户就需要额外判断这个标签到底对应哪个输入框。相反，如果标签，输入框和提示文字彼此靠近，用户很快就能够理解它们属于同一个操作单元。

所以设计中应该把相关的元素靠近，不相关的元素拉开。

接近性原则所要表达的重点是间距不是装饰，而是在表达关系。

### 2.2 相似性：长得像，就会被认为功能相近

相似性原则是指的是：外观相似的元素，会被用户认为具有相似功能或属于同一类别。

如果几个按钮颜色相同，形状相同，大小相同，用户会自然认为它们是同一种操作。反之，如果一个按钮用了完全不同的颜色，用户会自然认为它代表一种不同类型的操作。

所以子啊界面设计中，按钮样式，图标风格，字体大小，颜色规则都不能随意变化，以为每一次变化都是在向用户传递信号。

所以，保存，取消，返回这些普通操作一般都是相对统一的样式，但删除账号这种高风险的操作，就应该和普通按钮进行明显区分。

相似性原则所要表达的重点是：样式一致，是在帮助用户建立规则，而样式如果要发生变化，则应该有明确的理由。

### 2.3 共同区域：被框在一起，会被认为是同一组

共同区域原则是指的是：被同一个边框，卡片，色块或背景区域包裹的元素，会被用户认为是属于同一个功能组。

所以现在界面设计中经常会使用卡片，卡片不止是为了显得好看，也是在告诉用户：

> 这一块内容属于同一个主题 里面的按钮、文字、图片是相关的

codex写前端也经常会使用卡片，左边一块，右边一块的，但codex的卡片一般都会比较大，并且其中字体行距这些也比较大，codex默认会喜欢用那种又粗又硬的字体，很不符合我的审美，所以我后面就自己做了个skills。

当然举个更加简单的例子吧，一个商品卡片里会包含商品图，商品名，价格和购买按钮，但很明显你不会把这几个当成独立对象，而是会理解为“这是一个商品”

但共同区域不能乱用，如果一个界面到处都是边框，色块，卡片的话，用户可能会不知道重点在哪里，区域太多，视觉上就会变成碎片化的噪音。

共同区域原则想要告诉我们的核心内容是：边界可以帮助理解，但过多的边界会制造混乱。

### 2.4 闭合原则：大脑会主动补全缺失的信息

人的大脑会倾向于主动把一个不完整的的图形进行一个补全，把它看成一个完整形状。所以有些品牌的logo不选择画满所有边界，比如一个圆缺了一小段，人们依然会把它理解成一个圆。

闭合的价值在于可以让设计更加的简洁，有的时候适度的留白省略，反而会给用户产生特别的记忆点。

但闭合原则的前提在与闭合这个形状要非常的简单，如果缺失太多，比如一个圆缺少了99.9%，那都快成一条直线了，还补个der。

闭合原则告诉我们：省略可以制造美感，但用户必须要能够补得上。

### 2.5 简洁律：复杂不是问题，无序才是问题

在格式塔心理学的思想中还有一个很重要的思想，人的大脑倾向于把复杂信息组织成更简单，稳定，更有规律的形式。

这不意味着人只喜极简，只是说人的大脑更喜欢那种有规律的东西，有些建筑虽然很复杂，构成的部分很多，但是形成了一种别样的美，比如说很多欧洲中世纪的教堂，里面包含了非常多色彩纷呈的视觉元素，但它们不是随机堆砌的，而是通过重复，对称这些来形成秩序。人的大脑很享受这种从复杂中找到规律的感觉，所以也能产生一种美感，但前提是要让大脑能够在发现背后的规律。

加工流畅性理论也认为：一个对象越容易被加工，就越容易让人产生积极的审美反应。

所以用户其实并不怕复杂，怕的是没有规律的复杂。

## 三.认知负荷

当用户理解一个界面，做出一个选择，完成一个任务时，都要消耗一定的心理资源，或者说是心力吧。人的注意力和工作记忆都是有限的，一个界面如果信息太多，太复杂，用户就不得不把大量精力放在理解界面本身上，而不是花在“完成自己的目标”上。

所以这就是为什么像coze这样的产品功能并不少，但用起来很顺，有些产品功能也不复杂，却一打开就让人头疼。问题往往不在于功能数量，而在于有没有替用户组织好信息。

用户不应该被迫思考像这个按钮到底能不能点击，这些选项有什么区别，这个页面现在处于哪一步之类的问题。

### 3.1 信息太多会累

信息太多不只是界面上字很多，还包括颜色太多，按钮太多，图标太多，样式太多，提示太多。

当所有东西都在争夺用户注意力时，用户就不知道应该先看哪里。一个页面如果同时有横幅，弹窗，红点，广告，按钮，推荐内容和复杂表单，用户往往看到的不是丰富，而是噪音很多。

清晰的视觉层级可以引导用户看向页面中最重要的元素，可以通过颜色，对比，尺寸和分组来建立。也就是说，设计不是把所有信息平铺出来，而是告诉用户：什么最重要，什么次重要，什么可以稍后再看。

当然这样说的话可能会有些抽象，举个现成的例子，Apple的网站设计中iphone的产品页。

![](/uploads/1779522496473-f6y16s.png)

可以看到该页面把iphone根据具体产品拆成了几个明确模块，比如IPhone17 Pro，每个模块下面都有一句很短的卖点文案，再放Learn more和Buy两个操作入口。

1.用尺寸告诉你先看产品本身

首先在Iphone界面中，产品图和产品名是最大的视觉元素，用户第一眼看到的不是参数表，法律说明，而是产品本身，这就是尺寸的作用：“越大，越像主角”。

2.用短文案告诉你这个产品主打什么

可以看到这里的文案如The thinnest iPhone都是很短的，不是把所有参数一次性塞给你，而是先给你一个核心的印象。

3.用分组告诉你这一块属于同一个产品

把产品名，买点，按钮都放到同一个区域里，用户自然会理解哪些信息属于iphone 17pro，哪些属于iphone 17。

比较差的设计是把所有信息都显示出来让用户自己找重点，比较好的是当前步骤只显示当前最需要的信息，其余信息放到用户需要时再展开，skills中的渐进式披露也是类似于这样。

### 3.2 选择太多会太慢

在生活中选择过多往往会导致我们难以选择，毕竟很多情况下我们不能既要又要。从而可以引出在设计里，选择太多往往会带来另一个问题：“用户决策变慢，甚至干脆不选。”

在心理学中有个叫做希克定律的定律，描述的是反应时间与可选择项数量之间的关系，其实很简单，希克定律就是在说：

> 选项越多、信息越不确定，人做决定通常越慢。

如果放到产品设计中，当用户面对太多同等重要的选项时，需要先比较，理解，排除然后再做决定。

比如一个页面上如果同时有：

- 立即购买
- 加入购物车
- 收藏
- 对比
- 分享
- 对比
- 查看详情
- 领取优惠
- 咨询客服
- 分期付款
- 加入会员

如果这些功能在视觉上都差不多重要的话，用户就会产生犹豫。

更好的做法是建立决策层级：

- 主要行动：立即购买
- 次要行动：加入购物车
- 辅助行动：收藏，对比，咨询

像很多购物APP的商品界面大致就是这么个呈现方法，这样用户一眼就能知道主要路径是什么，而不会被一堆按钮给拖住。

## 四.目标梯度效应

目标梯度效应的核心是：“人越接近目标，越愿意继续投入”。大多数人没有正反馈就很难坚持下去，就反映了目标梯度效应。比如在抽卡游戏中，如果还差一张卡就能补齐，人会更想继续抽。以及如果在资料填写界面，显示用户已经完成了80%的填写，用户也会更愿意把剩下的部分填写完成。

所以很多时候比如像生成，加载这些界面如果只有一个无限旋转的圈圈，这个设计其实并不是很好，其实更好的设计是显示加载的进度，或者显示当前处于哪一步。当然也不是说那种循环的都不好，比如PVZ2的加载动画是一个金盏花，花瓣的数量会随着进度一点点变化，总体还是一个圆环的样式。

### 4.1 进度条

进度条是目标梯度效应最常见的设计形式。

当用户进入游戏界面，导出视频，安装软件，进度条可以很清晰的告诉用户当前的进度到了什么位置，相比一个不断转圈圈的设计，看起来更安心，更踏实。

比如一个进度条显示正在下载插件72%，比一句简单的下载中，其实更让人愿意去进行等待。

### 4.2 任务清单

任务清单是另一个目标梯度效应的常见应用，一个大的任务，如果不拆分，直接放在用户面前，其实是很让用户为难的，比如说填写账号资料。像很多银行的软件，在填写账号资料这一个任务时，都在上面把子任务拆分的明明白白，比如填写电话，身份证这些。这样用户明显就更愿意去行动，每完成一项，用户都会获得一个小的完成感。

任务清单可以把大而模糊的任务转化为小而明确的推进。

### 4.3 加载状态

加载状态可以分为几种：

短等待：可以使用简单loading动画，像那种只需要等待几秒钟的可以采用这种处理方式。

中等待：告诉用户系统正在做什么

长等待：提供进度，阶段，最好把预计时间也加上，或者允许用户离开后等通知。比如说像多智能体处理这种，我觉得3分钟以上其实对大多数用户就算长等待了。

## 五.微文案设计

积土成山，风雨兴焉，积水成渊，蛟龙生焉。很多时候，我们往往容易关注一些大的分块，布局，但很容易忽略一些小细节，比如说字体或文案。

乔布斯曾经强调，即使是用户看不见的地方，也应该认真对待。有一方面是收到他的工人父亲的影响，不过这真的是一个很不错的品质。很多时候，影响用户体验的，不一定是某个大的业务流程，界面排版，有可能是按钮上的几个字、表单上的一个提示、出错时的一段说明、空白页里的一段话。

这些容易被很多产品经理和开发者忽略的东西就是微文案。

微文案不是把文字搞可爱一点，也不是随便加点提示，微文案的作用是在关键时刻帮助用户理解状态、减少困惑、降低心理压力，并引导用户完成下一步。

### 5.1 按钮文案

一个好的按钮文案应该回答一个问题那就是“用户点击后到底会发生什么？”。

所以按钮文案的重心不在短，而在于明确。很多产品的按钮文案喜欢用“确定”，“提交”，“下一步”，这些词在很明确用户知道做什么的场景下适合，但有的时候用户并不知道自己点下去会发生什么。

比如一个提交按钮，到底是提交啥呢，如果是提交申请那我就知道了，提交的是申请。

再举两个例子：

```
差：确定
好：删除项目

差：完成
好：保存设置
```

由此可见好的按钮尽量使用动作+对象的结构，这样的结构下用户不太需要过度猜测按钮的含义。

但是这里需要一个特例，如果是人文关怀类的产品，按钮文案还需要注意降低压力。

比如普通产品可以写“开始测试”，但心理类产品更适合的是“开始了解自己”/“从一个小问题开始吧”

因为用户进入这种人文关怀类的产品，可能本来就带有一些负面情绪，如果按钮太标准化，像考试，诊断一样，就容易让用户紧张。

所以，心理类产品的按钮文案要尽量避免：

```
立即测评你的心理问题
开始诊断
检测异常状态
```

更温和的写法是：

```
了解当前状态
记录此刻感受
从一分钟记录开始
```

按钮文案的核心是：

> **让用户知道自己要做什么，同时不要让他觉得自己正在被审判。**

### 5.2 表单提示

表单提示的作用就是告诉用户在出错之前应该如何写

比如说一个很僵硬的表单内容如下：

```
请填写昵称
请填写手机号
请填写介绍
```

但这样会出现一些问题，比如：

1. 我填了昵称发现需要买SVIP才能修改姓名，居然不提前告诉我，曹贼啊.....
2. 我好不容易写完1k字的自我介绍，结果提交时告诉我自我介绍限制字数200，导致恨不得把吃的饭打翻在桌子上

这些问题虽然都是些小问题，但如果存在肯定会有人遇到，毕竟小概率事件只要次数范围足够大，除非概率是0，否则可以说是必然事件。

更好的表单提示应该在必要时补充格式、用途和限制。

比如说刚刚的那个可以改成下面那样：

```
昵称会显示在你的个人主页，之后可以修改/需要SVIP才能修改
请输入 11 位手机号，用于接收登录验证码
请描述你现在的感受，限制500字以内
```

当然如果是人文关怀类产品需要更细致一点，至于我这里为啥梅开二度了是因为我觉得下行时代，这种产品的需求是绝对会提高的，人们需要人文关怀或者说心理健康类的产品。

比如说一个心理记录页面，你可以只写一句：

```
请描述下你的情绪
```

也可以考虑更周到一点，毕竟用户可能压力很大，以至于不知道要写啥，写多少

```
现在的感觉可能很难准确描述，写几个词也可以。
```

或者

```
不需要整理得很清楚，先把此刻想到的写下来就好。
```

让用户感觉到自己在这里是被允许被接纳的

- 允许自己不完美
- 允许慢一点
- 允许只写一点点

### 5.3 错误信息：不要只说错误，还要把原因带上

很多时候尤其是AI大模型在编写代码的时候，对报错信息，可能前端就看到个error，连具体发生了啥都没有显示，或者说更简单的，直接返回个简单的“发生错误”。可能开发人员尚能看懂这是啥，但如果是那种小学生呢，可能还以为自己触发啥报警了呢。

当然还有稍微好一点的，比如：

```
xxx参数错误
操作无效
系统繁忙
提交失败
```

这些倒是像人话了，但我明明写的好好的东西，突然告诉我个提交失败，这是啥子道理，只告诉用户失败了，没告诉用户如何解决。

这上面最简单的，系统繁忙的话提醒用户休息一会等会来呗。

文件上传失败一般的原因是大小过大，把原因也告诉用户告诉用户应该控制在多大范围内。

所以可见一个好的错误提示应该说明3件事：

```
发生了什么？
为什么无法继续？
用户现在可以怎么做？
```

在开发时候可以显示程序员应该如何做，比如一大串英文的报错信息，看了就知道怎么改了。不知道的话直接上吧GPT5.5，使用bug修复，框框修好交给你，当然最好还是审查下，指不定埋啥坑了。但这些东西尽量不要暴露给用户。

这里我们尽量从用户的视角来进行考虑，这里我们继续谈下人文关怀类的产品，极端点，用户操作错误了，直接告诉用户：“你输入的内容无效”，或者说更极端点，假如写代码的AI或程序员没人性吧，直接显示：“哎呀，你咋这都能填错呢”。

这种很明显就容易引发用户的挫败感，哎，我怎么这么废，或者更严重的，顺着网线或者社会工程学爬过去把开发殴打一顿，谁叫你这么写的，咋还能这么泯灭人性呢？

### 5.4 空状态

空状态是很多产品最喜欢偷懒的地方，比如很多产品可能会写

```
暂无数据
暂无日记
暂无内容
...
```

这类文案其实是有点冷的，空状态应该用来传达系统状态，提高用户对系统的理解，提供可以通往关键任务的直接途径，尽量告诉用户可以做什么，怎么做？

举个例子，比如一个界面显示暂无项目，用户没注意右上角的+键，然后就会奇怪，对啊，我是没创建，但我去哪里创建呢？
比较好的是：你还没有创建项目，点击右上角新建项目，开始创建第一个项目。

再举两个例子:

```
差：暂无记录
好：这里还没有记录。等你准备好了，可以写下第一句话。

差：暂无搜索结果
好：没有找到相关内容。可以换个关键词再试试。
```

## 5.5 成功反馈：告诉用户努力已经生效

大多数的成功反馈都是成功，操作完成，已完成，仿佛在说这段故事已经结束了。

但这些话虽然能表达状态，但缺少信息。用户可能还想知道：

> 成功之后会发生什么？
> 
> 设置什么时候生效？
> 
> 内容保存到哪里了？
> 
> 我接下来要不要继续操作？

更好的成功反馈应该告诉用户结果和下一步。

比如：

```
差：保存成功
好：设置已保存，将在下次启动时生效。

差：发布成功
好：文章已发布，可以在个人主页查看。

差：提交成功
好：申请已提交，我们会在 1–3 个工作日内通知你结果。
```

在心理类产品中，成功反馈可以稍微多一点鼓励，但不要过度煽情。

比如：

```
记录已保存。
谢谢你把这段感受留在这里。
```

或者：

```
今天的记录完成了。
你已经为自己停留了一小会儿。
```

这种表达比简单的“保存成功”更有温度，但也不会过度承诺。

注意，心理类产品不要轻易写：

```
你已经战胜了焦虑
你一定会好起来
你的问题已经解决
```

这些话过度乐观，也可能带来不真实感。更稳妥的方式是承认一个小进展：

```
你完成了这次记录。
这不代表问题马上解决，但它是一个开始。
```

成功反馈的核心是：

> **让用户知道自己的行动产生了结果，并且这个结果是安全、可理解、可继续的。**
