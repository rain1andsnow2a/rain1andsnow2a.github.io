现在感觉系统架构思维是越发的重要了，随着AI的发展，想要做一个应用的难度是真的大打折扣了，甚至随便一个人加上claude opus4.7或者gpt5.5都能做一个以前需要全栈开发才能做出的应用。

<p>&nbsp;</p>

不过AI从某种程度上只是让功能的实现变得容易了，AI很擅长做局部的功能，比如一个登录功能等。

<p>&nbsp;</p>

但一个应用里面往往是很多个功能的集合，这些功能又会存在很多的关系，甚至于很多的流程在一起相互交错，如果只是让AI一个个做功能的话，很容易一个错误带崩一片，最后写出来的东西问题也会比较多，比如：

```
1.功能各写各的
2.数据模型前后不一致
3.本来可以封装成一个函数的东西重复写了好几次
4.权限边界不清晰
5.一个改动带崩一片
```

虽然说模块化编程能减少很多问题，但模块化从某种程度上来说也只是把代码拆开了而已，还是在形而下的代码层。

> 没有架构约束的模块化，也只是把混乱拆开了而已，从单个来看不是很乱，但无法改变整体的乱。

<p>&nbsp;</p>

所以问题来了，如何让AI写出更加符合实际业务的程序呢，需要框架的约束。如果说go,tauri,rust,postgresql,ACP协议这些是技术以及技术栈，那技术和技术栈它属于一种形而下的比较好把握的东西，某个功能也是一样。但架构属于那种形而上，需要把一个业务抽象成一个个系统，以及一些核心对象，是一个比较抽象的东西，但我们需要用形而上的架构去约束形而下的技术。

## 一.什么是架构

从定义上，架构是面对复杂需求时，对系统边界；模块关系；数据流；控制流；约束；演化路径做出的关键决策集合。其实我一开始看到这个定义的时候是有点懵的，系统边界是什么，控制流，演化路径又是什么？

<p>&nbsp;</p>

但从根本的角度，或者说架构设计的目的，这里分两种，一种是系统架构，一种是产品架构，产品架构更偏向于用户角度，系统架构更偏向于功能。

产品架构：产品的能力如何组织，用户的需求是否能够成立

系统架构：系统如何分工，数据怎么流动，如何约束

并且系统架构往往会有好几种方案，能够说明为什么方案A比方案B要好也是个很重要的能力，并且这往往需要一些更底层的理解。

## 二.架构应该从需求中来

从需求推导出系统的架构是一个比较重要的能力，我之前做开发往往是在堆砌模块，比如说如果用户的需求是做一个AI学习助手，那我可能会堆上AI答疑，智能路径规划，PDF上传解析，错题本，每日打卡，积分系统，积分商店，社区系统...，但很容易就变成这些功能之间没有稳定的结构或者关系，最后变成了一堆杂烩，每个模块各做各的。

但如果是架构思维的话，往往不会先考虑后端用java还是go，用postgresql还是mysql，用qdrant还是chromdb，而是先分析用户的需求是什么，需求反映的痛点是什么？

比如用户的需求是我想学的更好，这个例子可能比较宽泛了，甚至可以说是一个很差的prompt，但是可以试着从里面猜一下用户真正的痛点是什么：

```
1.不知道自己应该学哪些东西
2.不知道自己适合什么样的学习方法
3.学完没有反馈，或者说缺乏正反馈
4.今天学一点，明天忘一点，没有内化形成自己的一套东西
5.不知道哪些课程，文章，资料适合自己
```

所以用户要的不是一个有社区，有积分系统，有AI答疑的机器人

而是一个可以真正帮助自己解决缺乏正反馈，规划学习路径，推荐学习资源，复盘巩固，个性化的个人操作系统。

接着就可以去推导系统应该具有哪些架构能力

```
1.个人画像描绘能力
2.遗忘曲线管理能力
3.反馈调整能力
4.资源推荐能力
5.复盘沉淀能力
...
```

## 三.什么是系统思维呢

在开发一个产品时，通常把一个业务场景以整体论的方法把它抽象出来，叫做一个系统，接着就要去拆出系统中的主要对象。

或者我觉得这个方法有点像第一性原理，先把事物拆到基本的主要构成部分，再评估每一项是否是可以被舍弃的，由剩下的真理进行重构。

比如说一个外面平台，如果只看功能的话，我们会看到：

```
搜索店铺
浏览菜品
加入购物车
下单
支付
配送
评价
```

但使用系统思维，我们就能超出功能，看到更底层的对象与对象之间的关系

```
用户创建订单
订单包含彩票
订单绑定商家
订单触发支付
订单进入配送
配送分配骑手
订单完成后产生评价
异常情况下订单可能会退款，取消，超时或投诉
```

所以可以得到外卖系统的核心对象清单有

- 用户
- 商家
- 菜品
- 订单
- 支付
- 骑手
- 评价
